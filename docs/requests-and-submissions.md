# Requests And Submissions

## Purpose

Provide the public-facing intake model for improvements, examples, and
contributions across the family.

## Start With The Right Lane

Use one of these lanes:

- learn
- build
- contribute

### Learn

Use this lane for:

- missing lessons
- unclear onboarding
- weak explanations
- course or pathway gaps

### Build

Use this lane for:

- working examples
- templates
- workflow ideas
- pathway case studies
- reusable docs improvements

### Contribute

Use this lane for:

- governed engineering changes
- contract changes
- implementation proposals
- repo-boundary and architecture changes

## Rule

The public docs should explain how to enter the family, but they are not the
governed review ledger by themselves.

Requests and submissions should move toward the repo or process that currently
owns the work.

## Good Public Contributions

- lesson drafts
- onboarding fixes
- examples and walkthroughs
- template improvements
- better family maps
- pathway proposals for `uHOME`, Wizard, or Sonic

## Ownership Reminder

- concept and pathway improvements usually belong in `uDOS-docs`
- runtime and contract work belongs in the repo that owns that behavior
- private app implementation does not belong in the public family docs
