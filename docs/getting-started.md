# uDOS-docs Getting Started

1. Start with repo READMEs and boundary docs across the family.
2. Read `architecture/07_family_learning_path.md` for the current family ladder.
3. Review `architecture/05_v1_archive_relevance_assessment.md` and
   `architecture/06_v1_archive_asset_migration_matrix.md` before promoting archive material.
4. Review `architecture/08_seed_and_template_policy.md` before promoting
   archived seeds or examples.
5. Review `docs/activation.md`.
6. Run `scripts/run-docs-checks.sh`.
7. Keep naming aligned with code and public contracts.
8. Add examples only when they clarify a stable concept.
9. Use plain language before internal shorthand.
10. Prefer active public docs over archived wiki pages when both exist.
